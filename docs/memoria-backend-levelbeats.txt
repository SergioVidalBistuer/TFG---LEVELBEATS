DESARROLLO BACKEND DE LEVELBEATS - BORRADOR PARA MEMORIA TECNICA
=================================================================

A) RESUMEN BACKEND DEL PROYECTO
-------------------------------

El backend de LevelBeats está desarrollado con Laravel y PHP, siguiendo una arquitectura organizada por rutas, controladores, modelos Eloquent, middlewares, servicios y clases auxiliares. La aplicación funciona como una plataforma musical que integra marketplace de beats y colecciones, contratación de servicios técnicos, proyectos entre clientes e ingenieros, mensajería, gestión de roles profesionales, compras, licencias, facturas PDF, perfiles públicos, analíticas y administración.

La lógica principal del sistema reside en `app/Http/Controllers`, donde cada controlador se encarga de un módulo funcional concreto. Los datos se representan mediante modelos ubicados en `app/Models`, cada uno vinculado a tablas de la base de datos como `usuario`, `beat`, `coleccion`, `servicio`, `compra`, `factura`, `proyecto`, `suscripcion`, `rol`, `usuario_rol`, `conversacion`, `mensaje_directo` o `guardados`.

El backend no se limita a operaciones CRUD. Incluye reglas de negocio específicas: publicación u ocultación de productos, selección de licencias, validación de compras, exclusividad de licencias, generación de snapshots de compra, gestión de facturas, control de propiedad, activación de roles profesionales, pago simulado de planes, seguimiento de proyectos, subida de archivos, acceso a documentos PDF y cálculo dinámico de analíticas.


B) ARQUITECTURA LARAVEL Y PATRON MVC
------------------------------------

LevelBeats aplica el patrón MVC de Laravel de forma clara:

- Las rutas se definen principalmente en `routes/web.php`.
- Los controladores reciben las peticiones HTTP, validan datos, consultan modelos, aplican reglas de negocio y devuelven vistas o redirecciones.
- Los modelos Eloquent representan las entidades principales de la base de datos y sus relaciones.
- Las vistas Blade, aunque pertenecen al frontend, son renderizadas por el backend con los datos preparados por los controladores.
- Los middlewares controlan el acceso a rutas protegidas.
- Los servicios y clases de soporte encapsulan lógica reutilizable.

Las carpetas backend más relevantes son:

- `app/Http/Controllers`: controladores por módulo.
- `app/Models`: modelos Eloquent.
- `app/Http/Middleware`: middleware de login y administración.
- `app/Services`: servicios independientes, como generación de facturas PDF.
- `app/Support`: clases auxiliares para carrito y licencias.
- `routes/web.php`: rutas web públicas, privadas y administrativas.
- `storage`: almacenamiento de archivos subidos y documentos generados.

El proyecto utiliza modelos con nombres adaptados a una base de datos en singular, por ejemplo `Usuario` usa la tabla `usuario`, `Compra` usa `compra` y `Factura` usa `factura`. En varios modelos se desactivan los timestamps automáticos porque las tablas usan campos propios como `fecha_compra`, `fecha_emision`, `fecha_registro` o `fecha_creacion`.


C) ORGANIZACION DE RUTAS
-----------------------

La organización de rutas en `routes/web.php` se divide en tres grandes niveles:

1. Rutas públicas.
2. Rutas protegidas por login.
3. Rutas protegidas para administración.

Las rutas públicas incluyen:

- Inicio.
- Búsqueda global.
- Perfiles públicos.
- Contacto.
- Login y registro.
- Google OAuth.
- Catálogo público de beats.
- Catálogo público de colecciones.
- Catálogo público de servicios.
- Páginas legales.

Las rutas protegidas por `requirelogin` agrupan las funcionalidades de usuario autenticado:

- Onboarding de roles.
- Gestión de planes.
- Analíticas.
- Carrito.
- Checkout.
- Compras.
- Facturación.
- Encargos del usuario.
- Mensajería.
- Archivos de proyecto.
- Perfil y ajustes.
- Mis productos.
- Guardados.
- Studio.

Dentro del grupo protegido existe un subgrupo `studio`, organizado por prefijos:

- `studio/beats`.
- `studio/colecciones`.
- `studio/servicios`.
- `studio/proyectos`.

Las rutas administrativas están protegidas por `adminonly` y concentran:

- Dashboard admin.
- Gestión de usuarios.
- Beats.
- Colecciones.
- Proyectos.
- Servicios.
- Pedidos/compras.

Esta separación permite distinguir claramente qué partes son públicas, qué partes requieren sesión y qué partes solo puede usar un administrador.


D) MODULOS FUNCIONALES PRINCIPALES
----------------------------------

1. Autenticación y usuarios
~~~~~~~~~~~~~~~~~~~~~~~~~~~

El módulo de autenticación está gestionado por `AuthController`. Permite:

- Mostrar formularios de login y registro.
- Validar credenciales.
- Crear usuarios.
- Iniciar sesión con Laravel Auth.
- Cerrar sesión.
- Registrar usuarios con rol base.
- Acceder mediante Google OAuth usando Laravel Socialite.

El modelo `Usuario` adapta Laravel a una tabla propia llamada `usuario`, con columna de contraseña llamada `contrasena`. Para ello implementa métodos específicos como `getAuthPassword` y desactiva el remember token porque la tabla no lo contiene.

El `UsuarioController` gestiona la cuenta del usuario:

- Perfil.
- Perfil público heredado.
- Ajustes de datos personales.
- Foto de perfil.
- Cambio de contraseña.
- Biblioteca de productos comprados.
- Descarga de beats comprados.
- Generación de licencias PDF de productos comprados.
- Gestión administrativa de usuarios.

2. Roles, planes y suscripciones
~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~

El sistema de roles se basa en las tablas `rol` y `usuario_rol`. Un usuario puede tener varios roles activos al mismo tiempo. El modelo `Usuario` define la relación `roles()` y métodos auxiliares como:

- `tieneRol()`.
- `esAdmin()`.
- `tieneSuscripcionActiva()`.

Los roles relevantes son:

- Usuario básico.
- Productor.
- Ingeniero.
- Administrador.

El onboarding inicial está gestionado por `OnboardingController`. Permite seleccionar el camino del usuario y elegir planes asociados a productor o ingeniero.

La gestión posterior de roles y planes está centralizada en `PlanGestionController`. Este controlador:

- Muestra roles y suscripciones actuales.
- Redirige a selección de plan para activar rol profesional.
- Cancela roles profesionales sin borrar histórico.
- Muestra resumen de pago de plan.
- Valida que el plan pertenece a productor o ingeniero.
- Confirma el alta del plan.
- Cancela suscripciones anteriores del mismo rol.
- Crea una nueva suscripción activa.
- Activa el rol profesional en `usuario_rol`.
- Registra compra y factura en planes de pago mediante un flujo de pago simulado.

El pago de planes está preparado como simulación. Los planes gratuitos se activan tras confirmación; los planes de pago generan registro en `compra` y `factura`, manteniendo trazabilidad económica y dejando preparada una futura integración con una pasarela real.

3. Marketplace
~~~~~~~~~~~~~~

El marketplace está dividido en beats, colecciones, servicios y perfiles.

`BeatController` gestiona el catálogo público de beats:

- Listado público.
- Filtros por título, género, BPM, tono, estado de ánimo y precio.
- Detalle de beat.
- Comprobación de visibilidad pública.
- Carga de licencias disponibles.
- Comprobación de licencia exclusiva vendida.
- Comprobación de guardados.

`ColeccionController` realiza una función equivalente para colecciones:

- Listado público.
- Filtros por título, tipo, género, precio y número de beats.
- Detalle.
- Relación con beats incluidos.
- Guardado de portada.
- Validación de tipos y géneros.
- Sincronización de beats asociados.

`ServicioController` gestiona servicios públicos de ingenieros:

- Listado de servicios activos.
- Filtros por título, tipo, precio, plazo y revisiones.
- Detalle de servicio.
- Inicio de contacto con el ingeniero mediante creación de proyecto.

`PerfilController` gestiona el directorio público de perfiles, filtrando usuarios con perfil público y rol profesional activo.

`SearchController` centraliza la búsqueda global en beats, colecciones y servicios visibles públicamente.

4. Carrito y compras
~~~~~~~~~~~~~~~~~~~~

El carrito está gestionado por `CarritoController` y por la clase auxiliar `App\Support\CarritoCompra`.

`CarritoCompra` normaliza la estructura del carrito en sesión y permite manejar:

- Beats con licencia.
- Colecciones con licencia.
- Servicios asociados a un proyecto.

La clave del carrito incluye producto y licencia, lo que permite que un mismo producto pueda tratarse de forma diferenciada según la licencia elegida.

`CompraController` gestiona:

- Listado de compras.
- Detalle de compra.
- Pantalla de checkout.
- Procesamiento de compra.

Durante el checkout se validan:

- Método de pago.
- Datos de dirección.
- Carrito no vacío.
- Disponibilidad de productos.
- Que el usuario no compre sus propios productos.
- Que la licencia exclusiva no esté vendida.
- Que un servicio haya sido aceptado por el ingeniero antes de pagarse.
- Que un servicio no esté cancelado o ya pagado.

La compra se registra dentro de una transacción. Se crea una fila en `compra`, se relaciona con beats, colecciones o servicios mediante tablas pivot y, para beats/colecciones, se crea `compra_detalle` con snapshots de licencia, formato, derechos, precio base, suplemento y precio final.

5. Facturación y PDFs
~~~~~~~~~~~~~~~~~~~~~

La facturación está gestionada por:

- `FacturaController`.
- `FacturaPdfService`.
- Modelo `Factura`.

Al procesar una compra se crea una factura con número tipo `LB-000001`, base imponible, impuestos, total, estado de pago y fecha de emisión.

El PDF de factura se genera bajo demanda. Cuando el usuario solicita la factura:

- Se comprueba que sea comprador o administrador.
- Se genera el PDF si no existe.
- Se guarda en `storage/app/public/facturas`.
- Se actualiza `factura.url_factura_pdf`.
- Se devuelve como PDF inline al navegador.

`FacturaPdfService` usa DomPDF, construye las líneas de factura desde `compra_detalle` cuando existen y, si no, usa relaciones legacy con beats, colecciones o servicios.

Las licencias PDF de productos comprados se generan desde `UsuarioController`, usando DomPDF y la plantilla `pdf.licencia`. El acceso está protegido para que solo comprador o admin puedan ver o descargar el documento.

6. Servicios y proyectos
~~~~~~~~~~~~~~~~~~~~~~~~

Los servicios técnicos se publican por ingenieros y se convierten en proyectos cuando un cliente contacta.

El flujo principal es:

- El cliente entra en un servicio activo.
- Envía un mensaje inicial.
- `ServicioController@contacto` crea un `Proyecto`.
- Se crea un primer `Mensaje` asociado al proyecto.
- El proyecto queda en estado `pendiente_aceptacion_ingeniero`.

Después, el flujo continúa en `UsuarioEncargoController`:

- El ingeniero acepta el servicio.
- El proyecto pasa a pendiente de pago del cliente.
- El cliente acepta y pasa al checkout.
- Al completar el pago, el proyecto pasa a `en_proceso`.
- Antes del pago puede cancelarse.

`StudioProyectoController` permite al ingeniero gestionar sus proyectos desde Studio, actualizar estado y eliminar proyectos cancelados o cerrados. También controla la propiedad del proyecto verificando que el servicio pertenece al ingeniero.

7. Mensajería
~~~~~~~~~~~~~

LevelBeats tiene dos sistemas de mensajes:

- Mensajes de proyecto.
- Mensajes directos entre perfiles.

Los mensajes de proyecto están gestionados por `MensajeProyectoController` y se asocian a la tabla `mensaje`. Solo pueden participar el cliente, el ingeniero del servicio o un administrador.

Los mensajes directos están gestionados por `MensajeDirectoController` y usan las tablas `conversacion` y `mensaje_directo`. El sistema:

- Crea o reutiliza conversaciones únicas entre dos usuarios.
- Evita conversaciones consigo mismo.
- Lista conversaciones del usuario autenticado.
- Permite ver solo conversaciones donde participa.
- Marca mensajes del otro usuario como leídos al abrir la conversación.
- Valida longitud y contenido del mensaje.

El modelo `Conversacion` incluye relaciones con ambos usuarios, mensajes y último mensaje, además de métodos para comprobar participación y obtener el otro usuario.

8. Guardados
~~~~~~~~~~~~

El sistema de guardados está gestionado por `GuardadoController` y el modelo `Guardado`.

Permite guardar:

- Beats.
- Colecciones.
- Servicios.

La implementación usa relación polimórfica mediante `guardable_type` y `guardable_id`, con una lista blanca de tipos permitidos. El método de toggle crea o elimina el guardado según exista previamente, evitando duplicidades funcionales.

9. Analíticas
~~~~~~~~~~~~~

Las analíticas se calculan dinámicamente en `AnaliticaController`. No dependen de una tabla de métricas precalculadas.

Métricas generales:

- Compras realizadas.
- Total gastado.
- Beats adquiridos.
- Colecciones adquiridas.
- Servicios contratados.
- Conversaciones.
- Guardados.

Métricas de productor:

- Beats publicados y ocultos.
- Colecciones publicadas y ocultas.
- Ventas de beats.
- Ventas de colecciones.
- Ingresos.
- Ticket medio.
- Últimas ventas.
- Productos más vendidos.
- Guardados recibidos.

Métricas de ingeniero:

- Servicios publicados.
- Servicios activos/inactivos.
- Proyectos recibidos.
- Proyectos abiertos y cerrados.
- Ingresos por servicios.
- Plazo medio.
- Últimos proyectos.
- Servicios más contratados.

El controlador usa Eloquent y consultas `DB::table` cuando necesita agrupar ventas desde pivots como `beat_compra`, `coleccion_compra` y `servicio_compra`.

10. Contacto
~~~~~~~~~~~~

El formulario de contacto está gestionado por `ContactoController` y el mailable `ContactoRecibido`.

El backend valida:

- Nombre.
- Email.
- Asunto.
- Mensaje.
- Aceptación de privacidad.
- Honeypot antispam.

No se guarda información en base de datos. El mensaje se envía por email mediante SMTP configurado en `.env`. Si ocurre un error, se registra en logs y se devuelve un mensaje amigable.

11. Administración
~~~~~~~~~~~~~~~~~~

La administración está protegida por `AdminOnly`. Sus controladores principales son:

- `AdminDashboardController`.
- `AdminBeatController`.
- `AdminColeccionController`.
- `AdminProyectoController`.
- `AdminServicioController`.
- Funciones administrativas de `UsuarioController`.
- Vistas de compras/facturas accesibles con rol admin.

El panel admin permite supervisar usuarios, pedidos, servicios, proyectos, beats y colecciones. La administración se diferencia de Studio porque no se limita a los recursos del usuario autenticado, sino que permite una vista global de la plataforma.


E) VALIDACIONES Y CONTROLES
--------------------------

El backend realiza validaciones con `$request->validate()` en formularios y acciones críticas.

Ejemplos detectados:

- Login: email y contraseña obligatorios.
- Registro: nombre, email único y contraseña mínima.
- Perfil: nombre, email único, dirección, descripción y visibilidad pública.
- Foto de perfil: imagen JPG/PNG/WEBP con tamaño máximo.
- Contraseña: contraseña actual, nueva contraseña y confirmación.
- Contacto: nombre, email, mensaje, privacidad y honeypot.
- Mensajes directos: cuerpo requerido y máximo 2000 caracteres.
- Mensajes de proyecto: contenido requerido y máximo 1000 caracteres.
- Archivos de proyecto: archivo requerido, válido y máximo 50 MB.
- Beats Studio/Admin: audio, portada, precio, publicación y datos técnicos.
- Colecciones: tipo válido, género válido, precio, portada y beats asociados.
- Servicios: tipo, precio, plazo, revisiones y portada.
- Checkout: método de pago, dirección y coherencia del carrito.
- Planes: rol profesional válido, plan asociado a ese rol y método de pago en planes de pago.

Además de validaciones de entrada, existen controles de autorización:

- `abort(403)` cuando un usuario intenta gestionar recursos ajenos.
- Verificación de propietario en beats, colecciones, servicios y proyectos.
- Verificación de comprador para facturas, licencias y descargas.
- Verificación de participantes en conversaciones.
- Middleware `requirelogin` para zona privada.
- Middleware `adminonly` para administración.


F) LOGICA DE NEGOCIO CLAVE
--------------------------

1. Compra de beats y colecciones
~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~

El usuario añade un beat o colección al carrito junto con una licencia. La clase `CarritoCompra` normaliza el carrito y calcula los importes usando `LicenciaCompra`. Al confirmar el checkout, `CompraController` crea la compra, relaciona productos mediante pivots y guarda snapshots en `compra_detalle`.

Los snapshots permiten conservar el nombre del producto, licencia, formato, derechos y precio pagado aunque el producto o la licencia cambien posteriormente.

2. Gestión de licencias
~~~~~~~~~~~~~~~~~~~~~~~

`LicenciaCompra` define los tipos de licencia permitidos: básica, premium y exclusiva. También centraliza precios, formatos y derechos. La compra exclusiva se controla consultando `compra_detalle` para evitar futuras compras exclusivas del mismo producto.

3. Contratación de servicios
~~~~~~~~~~~~~~~~~~~~~~~~~~~~

Un servicio de ingeniero genera un proyecto cuando el cliente contacta. El proyecto pasa por estados como pendiente de aceptación del ingeniero, pendiente de pago, en proceso, revisión, entregado, cerrado o cancelado. El pago de servicio se tramita individualmente desde el checkout y, al completarse, se relaciona la compra con el servicio y el proyecto.

4. Generación de factura
~~~~~~~~~~~~~~~~~~~~~~~~

Cada compra pagada genera una factura. El PDF se genera bajo demanda mediante DomPDF y se guarda en storage. El acceso está limitado al comprador o administrador.

5. Gestión de roles y planes
~~~~~~~~~~~~~~~~~~~~~~~~~~~~

El usuario puede activar productor o ingeniero desde la gestión de planes. La activación real del rol solo se produce al confirmar un plan. Si el plan es de pago, se crea una compra y factura; si es gratuito, se crea la suscripción activa sin movimiento económico. Al cambiar de plan se cancelan suscripciones activas anteriores del mismo rol.

6. Mensajería
~~~~~~~~~~~~~

Los mensajes directos permiten comunicación entre usuarios/perfiles sin depender de un proyecto. Los mensajes de proyecto quedan vinculados al encargo de un servicio y solo son accesibles por cliente, ingeniero o admin.

7. Gestión desde Studio
~~~~~~~~~~~~~~~~~~~~~~~

Studio permite al productor gestionar beats y colecciones, y al ingeniero gestionar servicios y proyectos. Los controladores de Studio limitan el acceso por rol y propiedad, y controlan también límites derivados de la suscripción.

K) REDACCION SUGERIDA PARA LA MEMORIA
-------------------------------------

El backend de LevelBeats se ha desarrollado utilizando Laravel como framework principal y PHP como lenguaje de programación. La aplicación sigue una arquitectura basada en el patrón MVC, donde las rutas reciben las peticiones HTTP, los controladores aplican la lógica de negocio, los modelos representan las entidades de la base de datos y las vistas Blade reciben los datos preparados por el backend para su presentación.

La organización del backend se concentra principalmente en `routes/web.php`, `app/Http/Controllers`, `app/Models`, `app/Services`, `app/Support` y `app/Http/Middleware`. Esta estructura permite separar responsabilidades y mantener cada módulo funcional de la plataforma en componentes independientes. Las rutas públicas permiten acceder al marketplace, servicios, perfiles, contacto, autenticación y búsqueda. Las rutas privadas, protegidas por autenticación, agrupan funcionalidades como carrito, compras, facturación, mensajería, productos adquiridos, guardados, analíticas, gestión de planes y Studio. Por último, las rutas administrativas están protegidas mediante middleware específico y permiten la supervisión global de usuarios, compras, servicios, proyectos, beats y colecciones.

El sistema de usuarios está gestionado por un modelo propio, `Usuario`, adaptado a la estructura real de la base de datos. Este modelo extiende la autenticación de Laravel y ajusta campos no estándar como la contraseña almacenada en `contrasena`. La autenticación permite login clásico, registro, cierre de sesión y acceso con Google OAuth mediante Laravel Socialite. Tras el registro, el usuario pasa por un flujo de onboarding donde puede seleccionar el tipo de uso de la plataforma.

LevelBeats incorpora un sistema multirol basado en las tablas `rol` y `usuario_rol`. Un usuario puede actuar como usuario básico, productor, ingeniero o administrador. Los roles profesionales se complementan con planes y suscripciones, gestionados mediante las tablas `plan`, `plan_por_rol` y `suscripcion`. La activación de productor o ingeniero requiere seleccionar y confirmar un plan. Los planes gratuitos se activan tras confirmación, mientras que los planes de pago se registran mediante una compra y una factura, utilizando un flujo de pago simulado preparado para una futura pasarela real.

El marketplace se organiza en distintos módulos backend. Los beats se gestionan mediante `BeatController`, las colecciones mediante `ColeccionController`, los servicios mediante `ServicioController` y los perfiles públicos mediante `PerfilController`. Cada controlador se encarga de obtener los elementos visibles, aplicar filtros, controlar la visibilidad pública y preparar los datos necesarios para las vistas. Los productos ocultos no se muestran en el catálogo público, pero pueden seguir siendo gestionados por su propietario desde Studio.

El sistema de compra utiliza un carrito almacenado en sesión y normalizado mediante la clase auxiliar `CarritoCompra`. Esta clase permite manejar beats, colecciones y servicios, incluyendo la licencia seleccionada en el caso de productos musicales. La clase `LicenciaCompra` centraliza los tipos de licencia disponibles, sus precios, formatos y derechos. Al finalizar una compra, `CompraController` valida el carrito, evita compras inválidas, crea la compra, relaciona los productos mediante tablas intermedias y registra snapshots en `compra_detalle`. Estos snapshots permiten conservar la información histórica de la licencia y precio pagado aunque el producto cambie posteriormente.

La facturación se integra mediante el modelo `Factura`, el controlador `FacturaController` y el servicio `FacturaPdfService`. Cuando se completa una compra, se crea una factura con base imponible, impuestos, total y número de factura. El PDF se genera bajo demanda utilizando DomPDF, se almacena en `storage/app/public/facturas` y solo puede ser consultado por el comprador o por un administrador. De forma similar, las licencias de compra se generan como PDF personalizado desde `UsuarioController`, usando los datos de `compra_detalle`, la licencia adquirida y el producto comprado.

El módulo de servicios permite que un ingeniero publique servicios técnicos y que un cliente cree un encargo a partir de ellos. Al contactar con un servicio se crea un proyecto y un primer mensaje asociado. Posteriormente, el proyecto pasa por un flujo formal: aceptación del ingeniero, aceptación y pago del cliente, estado en proceso, revisión, entrega, cierre o cancelación. El pago del servicio reutiliza el checkout general, pero solo se permite cuando el ingeniero ya ha aceptado el encargo. Además, los proyectos incorporan mensajería interna y subida de archivos, con controles para que solo participen cliente, ingeniero o administrador.

La plataforma incorpora dos sistemas de comunicación. La mensajería de proyectos se vincula a encargos concretos y utiliza la tabla `mensaje`. La mensajería directa entre usuarios utiliza `conversacion` y `mensaje_directo`, permite iniciar conversaciones desde perfiles públicos, evita conversaciones duplicadas entre los mismos usuarios y marca mensajes como leídos al abrir el hilo. En ambos casos, el backend comprueba que el usuario autenticado tenga permiso para acceder al contenido.

El sistema de guardados permite que un usuario conserve beats, colecciones o servicios para consultarlos más adelante. Está implementado mediante una relación polimórfica en el modelo `Guardado`, con una lista blanca de tipos permitidos para evitar referencias no válidas. El controlador permite alternar entre guardar y eliminar un elemento, evitando duplicaciones funcionales.

Las analíticas se calculan dinámicamente en `AnaliticaController`. Para usuarios básicos se muestran métricas generales de actividad, como compras realizadas, gasto, productos adquiridos, conversaciones y guardados. Para productores se calculan datos de catálogo, ventas de beats y colecciones, ingresos, ticket medio y productos más vendidos. Para ingenieros se calculan servicios publicados, proyectos recibidos, proyectos abiertos o cerrados, ingresos por servicios y servicios más contratados. Estas métricas se obtienen a partir de las tablas reales de compras, productos, servicios, proyectos y relaciones intermedias.

El backend incluye medidas de seguridad en distintos niveles. Las rutas privadas requieren autenticación, las rutas administrativas requieren rol de administrador y los controladores comprueban la propiedad de los recursos antes de permitir acciones sensibles. También se validan formularios, archivos subidos, mensajes, compras, métodos de pago y selección de planes. El acceso a documentos, facturas, licencias, descargas, conversaciones y proyectos está protegido mediante comprobaciones específicas y respuestas 403 o 404 cuando corresponde.

En conjunto, el backend de LevelBeats está diseñado como una plataforma modular y orientada a procesos reales: publicación musical, contratación de servicios, compra con licencias, gestión profesional, comunicación entre usuarios, facturación y administración. La combinación de Laravel, Eloquent, middlewares, servicios y clases auxiliares permite mantener una separación clara de responsabilidades y facilita la evolución futura del sistema, por ejemplo para integrar una pasarela de pago real o ampliar las métricas de analíticas.
