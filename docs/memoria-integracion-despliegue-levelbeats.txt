Integración y despliegue de LevelBeats
======================================

1. Introducción
---------------

Este documento recoge la información técnica relativa a la integración y despliegue del proyecto web LevelBeats. El objetivo no es describir el frontend ni el backend como módulos independientes, sino explicar cómo se conectan las tecnologías que permiten ejecutar la aplicación: Laravel, PHP, Apache, MySQL, Docker, Composer, NPM/Vite, almacenamiento público, correo SMTP y servicios externos como Google OAuth.

La información está basada en los archivos reales revisados en el repositorio y en la configuración Docker externa localizada junto al proyecto Laravel. Cuando algún dato no se ha podido confirmar directamente en el repositorio, se indica expresamente.


2. Resumen general de integración
---------------------------------

LevelBeats está construido como una aplicación Laravel ejecutada sobre PHP. Laravel actúa como núcleo de la aplicación: recibe las peticiones HTTP, ejecuta rutas y controladores, consulta la base de datos mediante Eloquent/Query Builder, renderiza vistas Blade y gestiona servicios como sesión, validación, correo, almacenamiento y generación de documentos.

La ejecución local se apoya en un entorno Docker externo al directorio principal de Laravel. Este entorno define tres servicios principales:

- Un contenedor PHP con Apache, construido a partir de la imagen `php:8.5.2-apache`.
- Un contenedor MySQL 8.0.
- Un contenedor phpMyAdmin.

El servicio PHP/Apache expone la aplicación en el puerto local 8086. MySQL se expone en el puerto 3310 y phpMyAdmin en el puerto 8087. La aplicación Laravel se monta dentro del contenedor mediante volumen, de forma que los cambios del proyecto son visibles directamente en el entorno de ejecución.

La arquitectura local queda resumida así:

- Navegador -> http://localhost:8086/public/ o virtual host configurado -> Apache/PHP -> Laravel.
- Laravel -> conexión MySQL mediante variables `DB_*`.
- Laravel -> storage local/public para audios, portadas, facturas, licencias y archivos de proyecto.
- Laravel -> SMTP para formulario de contacto.
- Laravel -> Google OAuth mediante Socialite.

4. Docker y servicios locales
-----------------------------

El archivo `../proyectoLaravel_docker/docker-compose.yml` define tres servicios:

1. Servicio PHP/Apache
   - Se construye usando el `Dockerfile`.
   - Monta el proyecto Laravel en `/var/www/html`.
   - Expone el puerto `8086:80`.
   - Permite servir la aplicación desde Apache.

2. Servicio MySQL
   - Usa la imagen `mysql:8.0`.
   - Define credenciales de desarrollo en el compose.
   - Expone el puerto `3310:3306`.
   - Usa un volumen persistente llamado `mysqldata`.

3. Servicio phpMyAdmin
   - Usa la imagen `phpmyadmin/phpmyadmin`.
   - Se conecta al servicio `mysql`.
   - Expone el puerto `8087:80`.

La imagen PHP se construye desde `php:8.5.2-apache`. El Dockerfile instala extensiones necesarias para Laravel:

- `pdo_mysql`
- `mysqli`
- `mbstring`
- `bcmath`

También habilita módulos de Apache:

- `rewrite`
- `autoindex`

El módulo `rewrite` es importante para Laravel porque permite que Apache redirija correctamente las peticiones hacia `public/index.php` usando las reglas de `.htaccess`.


5. Configuración de Apache
--------------------------

Se han localizado dos configuraciones de Apache:

- `config/apache.conf`
- `config/laravel.conf`

`apache.conf` define un VirtualHost general con `DocumentRoot /var/www/html`, permisos de acceso y `AllowOverride All`.

`laravel.conf` define un VirtualHost específico con:

- `ServerName laravel.local`
- `DocumentRoot /var/www/html/laravel/public`
- Directorio público de Laravel con `AllowOverride All`.
- Logs específicos de error y acceso.

Para una aplicación Laravel, el DocumentRoot correcto debe apuntar a la carpeta `public`, porque ahí se encuentra el punto de entrada `index.php` y los recursos públicos. En local también se ha indicado como URL habitual `http://localhost:8086/public/`, lo que sugiere que puede estar usándose el acceso directo por carpeta pública en lugar de un virtual host plenamente resuelto.

En una memoria técnica conviene explicar que Laravel debe servirse desde `public/` para no exponer archivos internos como `.env`, `app/`, `storage/` o `vendor/`.

8. Base de datos e integración con MySQL
----------------------------------------

La conexión con base de datos se configura mediante `config/database.php`, que utiliza variables de entorno:

- `DB_CONNECTION`
- `DB_HOST`
- `DB_PORT`
- `DB_DATABASE`
- `DB_USERNAME`
- `DB_PASSWORD`
- `DB_CHARSET`
- `DB_COLLATION`

Laravel tiene configuradas varias conexiones posibles: SQLite, MySQL, MariaDB, PostgreSQL y SQL Server. Para LevelBeats, el entorno Docker incluye MySQL 8.0, por lo que la configuración real esperada en `.env` debe usar conexión MySQL.

En Docker Compose se define:

- Host interno del servicio: `mysql`
- Puerto interno: `3306`
- Puerto externo local: `3310`
- Base de datos inicial: `daw`
- Usuario: `admin`

En producción estos valores deben adaptarse al servidor de base de datos proporcionado por el hosting o por el entorno elegido.

phpMyAdmin se utiliza como herramienta de gestión visual de la base de datos durante el desarrollo, accesible en local por el puerto `8087`.

10. Storage, archivos públicos y documentos generados
-----------------------------------------------------

Laravel utiliza `config/filesystems.php` para definir discos de almacenamiento.

Discos relevantes:

- `local`: apunta a `storage/app/private`.
- `public`: apunta a `storage/app/public`.

El disco público genera URLs bajo:

`APP_URL/storage`

La configuración incluye el enlace simbólico:

`public/storage -> storage/app/public`

Esto implica que para servir archivos subidos por usuarios o documentos generados debe existir el enlace creado mediante:

`php artisan storage:link`

Carpetas detectadas dentro de storage:

- `storage/app/public/beats`
- `storage/app/public/colecciones`
- `storage/app/public/servicios`
- `storage/app/public/usuarios`
- `storage/app/public/facturas`
- `storage/app/private/proyectos`
- `storage/app/licencias/referencia`

Uso funcional:

- Audios de beats.
- Portadas de beats, colecciones y servicios.
- Fotos de perfil.
- Facturas PDF.
- Archivos compartidos de proyectos.
- Plantillas/referencias de licencias.

Los archivos de proyecto se almacenan en disco local privado, mientras que portadas, audios y facturas usan principalmente el disco público.

En despliegue es importante asegurar permisos de escritura en:

- `storage`
- `bootstrap/cache`


11. Generación de PDFs
----------------------

El proyecto utiliza DomPDF, confirmado en `composer.json` mediante `dompdf/dompdf`.

Los PDFs se usan para:

- Facturas.
- Licencias/contratos de compra.

Las facturas se generan bajo demanda mediante `FacturaPdfService`. El PDF se guarda en:

`storage/app/public/facturas`

La URL pública se almacena como ruta tipo:

`storage/facturas/...`

Esto refuerza la importancia del enlace `public/storage` y de una configuración correcta de `APP_URL`.


12. Correo SMTP y contacto
--------------------------

El correo se configura en `config/mail.php`. Laravel usa variables de entorno para definir el mailer y los datos SMTP.

El formulario de contacto usa:

- `ContactoController`
- `ContactoRecibido`
- `Mail::to(...)`

La variable `CONTACT_MAIL_TO` se usa como destinatario principal del formulario. Si no existe, el código usa como fallback `MAIL_FROM_ADDRESS`.

Para Hostinger se ha previsto usar una cuenta de correo corporativo:

`contacto@level-beats.com`

Configuración esperada para producción:

- `MAIL_MAILER=smtp`
- `MAIL_HOST=smtp.hostinger.com`
- `MAIL_PORT=465`
- `MAIL_ENCRYPTION=ssl`
- `MAIL_USERNAME=contacto@level-beats.com`
- `MAIL_PASSWORD=...`
- `MAIL_FROM_ADDRESS=contacto@level-beats.com`
- `MAIL_FROM_NAME=LevelBeats`
- `CONTACT_MAIL_TO=contacto@level-beats.com`

No se deben incluir contraseñas reales en la memoria ni en el repositorio.


13. Google OAuth
----------------

`config/services.php` define integración con Google:

- `GOOGLE_CLIENT_ID`
- `GOOGLE_CLIENT_SECRET`
- `GOOGLE_REDIRECT_URI`

El login con Google está implementado mediante Laravel Socialite, dependencia confirmada en `composer.json`.

En local se ha usado como URL de callback:

`http://localhost:8086/public/auth/google/callback`

En producción será necesario registrar en Google Cloud una URI equivalente para el dominio definitivo, por ejemplo:

`https://level-beats.com/auth/google/callback`

La integración depende de que `APP_URL` y `GOOGLE_REDIRECT_URI` estén correctamente ajustadas al entorno.

16. Relación entre local y producción
-------------------------------------

En desarrollo local, Docker facilita reproducir un entorno completo en una máquina Ubuntu/WSL:

- Apache y PHP en contenedor.
- MySQL en contenedor.
- phpMyAdmin para gestión visual.
- Código Laravel montado por volumen.

En producción, Hostinger sustituye a Docker como entorno de ejecución final. La aplicación sigue necesitando los mismos elementos: PHP compatible, servidor web apuntando a `public`, base de datos MySQL, Composer, variables de entorno, storage escribible y SMTP operativo.

La diferencia principal es que en local los servicios se comunican por nombres de contenedor y puertos expuestos, mientras que en producción la conexión se realiza contra los servicios proporcionados por el hosting.

20. Redacción sugerida para memoria
-----------------------------------

La integración técnica de LevelBeats se ha planteado sobre una arquitectura basada en Laravel, PHP, Apache y MySQL. Laravel actúa como núcleo de la aplicación, gestionando las rutas, controladores, vistas Blade, modelos, servicios, almacenamiento, sesiones, validaciones y conexión con la base de datos. La aplicación se ejecuta en local mediante un entorno Docker que agrupa los servicios necesarios para reproducir el funcionamiento completo del sistema.

El entorno Docker está compuesto por tres contenedores principales. El primero contiene PHP con Apache, construido a partir de la imagen `php:8.5.2-apache`, e incluye extensiones necesarias para Laravel como `pdo_mysql`, `mysqli`, `mbstring` y `bcmath`. El segundo contenedor ejecuta MySQL 8.0 y almacena los datos de la aplicación en un volumen persistente. El tercero proporciona phpMyAdmin, utilizado como herramienta de administración de la base de datos durante el desarrollo. Esta organización permite separar la aplicación, la base de datos y la herramienta de gestión, manteniendo un entorno controlado y reproducible.

La aplicación Laravel se sirve a través de Apache. La configuración localizada incluye un VirtualHost específico que apunta al directorio `public` de Laravel, que es el punto de entrada correcto para este tipo de aplicaciones. Esto permite que las peticiones HTTP lleguen a `index.php` y sean procesadas por el sistema de rutas de Laravel. En desarrollo, la aplicación se expone en el puerto 8086, mientras que MySQL se expone en el puerto 3310 y phpMyAdmin en el puerto 8087.

La conexión con la base de datos se configura mediante variables de entorno. Laravel toma los valores de `DB_CONNECTION`, `DB_HOST`, `DB_PORT`, `DB_DATABASE`, `DB_USERNAME` y `DB_PASSWORD` para conectarse a MySQL. En local, estos valores deben ajustarse al servicio definido en Docker Compose. En producción, se sustituyen por los datos proporcionados por el hosting. Esta separación permite mantener el mismo código fuente para distintos entornos, modificando únicamente la configuración.

El proyecto utiliza Composer para gestionar dependencias PHP. Entre las dependencias principales se encuentran Laravel Framework, DomPDF para generación de documentos PDF y Laravel Socialite para autenticación con Google. En el lado de assets, el proyecto incluye Vite, Laravel Vite Plugin, Tailwind CSS, Axios y herramientas de desarrollo como Concurrently. Aunque la interfaz principal utiliza también CSS propio y Bootstrap, Vite queda disponible como sistema de compilación moderno para recursos frontend.

La gestión de archivos se realiza mediante el sistema de almacenamiento de Laravel. El disco público apunta a `storage/app/public` y se expone mediante el enlace simbólico `public/storage`. Este mecanismo se utiliza para servir portadas, audios, imágenes de perfil y facturas PDF. Por otro lado, algunos archivos de proyecto se almacenan en zonas privadas de `storage`, evitando su exposición directa. Para que el sistema funcione correctamente en producción, es necesario mantener permisos de escritura en `storage` y `bootstrap/cache`, además de crear el enlace simbólico con `php artisan storage:link`.

LevelBeats también integra servicios externos. El formulario de contacto utiliza el sistema de correo de Laravel, configurado mediante SMTP. Para producción se contempla el uso del correo corporativo de Hostinger `contacto@level-beats.com`, con las variables `MAIL_*` y `CONTACT_MAIL_TO`. Además, el login con Google se configura mediante `config/services.php` y las variables `GOOGLE_CLIENT_ID`, `GOOGLE_CLIENT_SECRET` y `GOOGLE_REDIRECT_URI`. En el despliegue final será necesario registrar en Google Cloud la URL de callback correspondiente al dominio de producción.

El despliegue previsto se orienta a Hostinger con el dominio `level-beats.com`. Para pasar de desarrollo a producción es necesario configurar el archivo `.env`, instalar dependencias con Composer, compilar assets si procede, configurar la base de datos, ajustar el correo SMTP, crear el enlace de storage, establecer permisos adecuados y optimizar cachés de Laravel. También es importante que el servidor web apunte a la carpeta `public`, ya que exponer la raíz del proyecto podría comprometer archivos internos como `.env`, `storage` o `vendor`.

En conjunto, la integración de LevelBeats combina un entorno local reproducible mediante Docker con una arquitectura Laravel preparada para producción. La separación entre aplicación, base de datos, almacenamiento, correo y servicios externos permite mantener una estructura ordenada, flexible y adecuada para el despliegue posterior en un hosting real.
